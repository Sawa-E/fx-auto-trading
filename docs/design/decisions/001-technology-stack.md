# 技術スタック選定

## 概要

fx-auto-trading v2 の技術スタックを選定する。予測モデルの精度最大化を最優先に、シンプルで保守しやすい構成を目指す。

## 設計判断

### 判断1: 言語 — Python 3.12

**問題**: 開発言語の選定
**選択肢**: 1. Python 2. Rust 3. Go
**決定**: Python 3.12
**理由**: ML/データサイエンスのエコシステムが圧倒的。LightGBM, pandas, scikit-learn 等のライブラリが成熟している
**トレードオフ**: 実行速度は劣るが、予測モデル開発ではボトルネックにならない

### 判断2: パッケージ管理 — uv

**問題**: Python パッケージ管理ツールの選定
**選択肢**: 1. uv 2. pip + venv 3. poetry
**決定**: uv
**理由**: 高速、pyproject.toml ネイティブ、lock ファイル対応。v1 から継続
**トレードオフ**: 比較的新しいツールだが、安定版に到達済み

### 判断3: 予測モデル — RF（ベースライン）+ LightGBM（メイン）

**問題**: 主要予測アルゴリズムの選定
**選択肢**: 1. LightGBM 2. XGBoost 3. RandomForest 4. LSTM (PyTorch) 5. SVM 6. ディープラーニング
**決定**: ランダムフォレストをベースライン、LightGBMをメインモデルとする
**理由**:
- 資料5（MUFG樋口）: 為替予測でRF→XGBoostの段階的プロセスで精度が大幅向上。「Kaggleで LightGBM と XGBoost が人気上位。ディープラーニングは万能ではない」
- 資料4（FFAJ岩壷）: RFの変数重要度分析で勝率>RR比>レバレッジ>保有期間の順位付けに成功
- 資料5: SVMは「下落中心の予測、変化の局面に反応できない」、DLは「2016年以降投資行動を取っていない」と為替で不良と結論
- 資料3（SIG-FIN）: LSTMを使用しているが共和分ペアトレードという特殊ケース。一般的な方向予測にはGBDT系が適切

**トレードオフ**: LSTMのような時系列固有の構造は捉えにくいが、適切な特徴量エンジニアリング（ラグ特徴量、移動平均等）で補える

### 判断4: データ取得 — GMO FX Public API + httpx

**問題**: FXデータソースの選定
**選択肢**: 1. GMO FX API 2. OANDA API 3. Yahoo Finance
**決定**: GMO FX Public API
**理由**: 日本国内FX業者で無料。USD/JPY の1時間足OHLCデータが取得可能。v1で実装実績あり
**トレードオフ**: 分足データの取得期間に制限がある

### 判断5: データストレージ — SQLite

**問題**: OHLCデータの永続化方式
**選択肢**: 1. SQLite 2. CSV/Parquet 3. PostgreSQL
**決定**: SQLite
**理由**: セットアップ不要、ファイルベースで可搬性が高い。個人利用には十分な性能
**トレードオフ**: 並行書き込みに弱いが、単一プロセスなので問題なし

### 判断6: 検証方式 — ウォークフォワード（拡大窓）

**問題**: モデル評価方式の選定
**選択肢**: 1. ウォークフォワード（拡大窓） 2. ウォークフォワード（固定窓） 3. 単純 train/test split 4. K-fold CV
**決定**: ウォークフォワード（拡大窓 = expanding window）
**理由**:
- 資料5（MUFG樋口）: 「1990年〜予測対象の1つ手前まで」の拡大窓方式で成功
- 資料3（SIG-FIN）: 22年分の固定学習期間（1985-2006）を使用。長い学習期間が有効
- 資料2（Optimax）: 「バックテストを過信するな。過学習が最大リスク」→ アウトオブサンプル検証が必須
- FXの1H足では固定窓だとサンプル不足になりやすく、拡大窓の方がサンプル数を確保できる

**トレードオフ**: 古いレジームのデータが残るが、LightGBMの正則化で対処。計算コストは高いが精度評価の信頼性には代えられない

### 判断7: HPO — Optuna

**問題**: ハイパーパラメータ最適化フレームワークの選定
**選択肢**: 1. Optuna 2. GridSearch 3. RandomSearch 4. 手動チューニング
**決定**: Optuna
**理由**: ベイズ最適化ベースで効率的。LightGBMとの統合が容易。枝刈り（pruning）で計算コスト削減
**トレードオフ**: 依存が1つ増えるが、チューニング効率の向上が大きい
